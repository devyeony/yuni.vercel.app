Petping is a pet-lifestyle app that manages walks, meals, health, insurance,
and shopping in one place. I joined as a backend engineer and, over two and a
half years, ended up owning most of the server side: API design, the AWS
infrastructure it ran on, and the operational tooling that kept it healthy.

## Building it

The domain grew fast — walk tracking with real-time events, meal logging,
insurance-claim submission against external insurers, and commerce
integrations. The persistence layer was the first structural decision: the
codebase was MyBatis-first, and rewriting it wholesale would have frozen
feature work for months. Introducing JPA for new domains while keeping MyBatis
for proven query-heavy paths gave us entity-level productivity where it paid
off, without betting the product on a migration.

Insurance claims were the deepest integration work: automating a previously
manual claim flow against external insurer APIs, with commerce API
integrations layered on top. The measure of success here wasn't a latency
chart — it was the customer-service queue, where insurance-related inquiries
dropped by a fifth.

## Operating it

A daily-use pet app can't take release downtime, so I built the CI/CD
pipeline up to blue/green zero-downtime deployment on AWS (ECS, CodeDeploy).
On the visibility side, a self-hosted Grafana + Loki stack streamed
structured logs into dashboards and alerts we tuned to our actual incident
patterns, cutting detection-to-response time by more than half. Redash views
gave the operations team self-serve statistics, and QA got dedicated test
APIs — both removed the backend team as a bottleneck from other people's
workflows.
