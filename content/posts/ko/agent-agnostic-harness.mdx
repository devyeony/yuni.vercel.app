이 레포에서 `CLAUDE.md`, `.cursor`, `GEMINI.md`를 grep하면 `.gitignore`
안에서만 발견된다. 사이트는 AI의 도움을 많이 받아 만들어졌지만, 커밋된
파일 어디에도 벤더 이름이 없다. 결벽이 아니라 설계다.

코딩 에이전트는 저마다 자기 지침 파일을 원한다. 그걸 커밋하면 같은 규칙이
도구마다 복제되어 서로 어긋나기 시작하고, 레포는 셋업하던 달에 유행하던
에이전트와 조용히 결혼하게 된다. 그래서 지침은 벤더 중립으로 한 번만
존재하고, 도구별 파일은 전부 생성되는 어댑터다:

```
AGENTS.md                    # 진입점 (크로스-에이전트 관례)
agents/
├── harness.md               # 루프: work → check → test → verify
└── skills/                  # 포터블 절차, 순수 마크다운
    ├── design-system/       # 토큰 3계층, 정확한 패키지명, CVA 패턴
    ├── code-review/         # 기계가 못 잡는 것 — 판단 기준
    ├── content/             # 경력·프로젝트·글 추가 절차
    ├── i18n/                # 로케일·메시지 절차
    └── verify/              # 루프 실행법과 해석법
scripts/setup-agents.mjs     # gitignore된 어댑터 재생성, 명령 하나
```

스킬 메커니즘이 있는 에이전트는 어댑터를 통해 네이티브로 로드한다. 없는
에이전트는 문서로 읽는다 — `AGENTS.md`가 경로를 링크한다. 도구 교체가
아무 일도 아닌 것, 그게 내가 원한 속성이다: 협업 구조가 특정 에이전트보다
오래 산다.

더 깊은 원칙: 지침은 시스템에서 가장 약한 계층이므로, 중요한 것은 지침에만
살도록 두지 않는다. 에이전트는 언젠가 세션 중간에 규칙을 잊는다 — 가능성이
아니라 기본 확률이다. 하네스는 그걸 전제한다. `pnpm check`는 Biome, strict
tsc, 데드 코드 탐지, en/ko 메시지 대칭 검사를 몇 초 안에 돌리고, `pnpm
verify`는 프로덕션 빌드 후 세 뷰포트에서 Playwright를 axe 스캔·시각
회귀와 함께 돌린다. CI는 동일한 명령을 재실행한다 — "로컬에선 됐는데"와
"CI에선 됐는데"가 같은 문장이 된다.

이 패턴의 내가 가장 좋아하는 사례: Open Graph 이미지 렌더러는 CSS 변수를
읽지 못해서 사이트 팔레트의 raw hex 사본이 필요하다 — 교과서적인
드리프트 위험이자 "시맨틱 토큰만 사용" 규칙의 정면 위반이다. 예외는
허용하되, 즉시 기계화했다: 단위 테스트가 `tokens.css`에서 hex 값을
재계산하고, 사본이 어긋나는 순간 빌드를 실패시킨다. 규칙이 자신의 예외보다
오래 살아남고, 사람이든 에이전트든 그걸 기억할 필요가 없다.

프롬프트는 소원이다. 루프가 계약이다.
