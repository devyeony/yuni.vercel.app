이 사이트의 모든 AI 기능 — 프로젝트마다 붙는 "비슷한 작업" 링크, 2D 임베딩
지도, 사이트 검색, MCP 툴 — 은 정적 파일 하나를 읽는다:
`public/rag-index.json`. 매 빌드의 선두에서 재생성되고, 다른 에셋과 똑같이
배포된다. 벡터 데이터베이스도, 임베딩 엔드포인트도, 런타임에 돌아가는
모델도 없다.

파이프라인은 스크립트 하나다. 모든 콘텐츠 컬렉션 — 프로젝트 케이스 스터디,
글, 경력, 활동 — 을 청크로 나누고, Transformers.js와
`multilingual-e5-small`(무료 양자화 모델)로 로컬에서 임베딩한다. 다국어
임베딩 공간 덕에 인덱스 하나가 영어와 한국어를 모두 커버한다. 현재 출력은
384차원 청크 34개, 약 125kB.

이 규모에서 "데이터베이스"는 배열이고, 쿼리 엔진은 fold 하나다:

```ts
const cosine = (a: number[], b: number[]) =>
  a.reduce((sum, v, i) => sum + v * b[i], 0);
```

여기에 벡터 데이터베이스를 두는 것은 한 줄짜리 코드를 피하려고 인프라를
운영하는 일이다. 인덱스 포맷이 이 선택을 격리하므로 — 코퍼스가 인메모리
검색을 넘어서면 검색 함수만 바꾸면 된다 — 개인 사이트가 그 규모에 도달하는
건 오히려 반가운 문제다.

가장 아끼는 부분: 인덱스는 스스로를 검증한다. 임베딩이 끝나면 빌드가 양
로케일로 스모크 검색을 돌린다 — 백엔드 경력을 묻는 영어 질의, 오픈소스를
묻는 한국어 질의 — 관련 청크가 top-3에 없으면 빌드가 실패한다. 깨진
인덱스는 배포될 수 없다. 배포 자체가 테스트라서.

하류는 전부 파생물이다. 프로젝트 파일 하나를 추가하면 케이스 스터디
페이지, RSS 피드, 유사 콘텐츠 링크, 지도, 검색, MCP 응답이 동시에 갱신된다
— 컴포넌트 수정은 0. 추천 블록은 코사인 점수를 일부러 노출한다. 메커니즘
자체가 보여줄 내용이니까.

정직한 한계: 검색은 하이브리드지, 완전한 시맨틱이 아니다. 렉시컬 매칭이
의사(pseudo) 쿼리 벡터의 발판이 되고, 그 벡터가 코사인으로 전체 코퍼스를
랭킹한다 — 렉시컬 발판이 없는 질의는 빈 결과를 낸다. 대안들은 실측 후
기각했다: 브라우저에서 모델을 돌리면 방문자의 첫 검색이 약 129MB를
다운로드하고, 서버리스 임베딩 라우트는 콜드 스타트가 3~8초다. 34청크
코퍼스에서 얻는 재현율은 미미한데, 대가는 내가 가장 아끼는 속성 — AI
레이어 전체가 정적이고, 무료이고, 런타임에 실패할 수 없다는 것 — 이었다.

이 사이트에는 원래 RAG 챗봇이 계획돼 있었다. 증명 가능한 쪽은 검색(R)이고
리스크는 생성(G)이라는 걸 알아챘을 때 잘라냈다. R은 지금도 모든 페이지에서
눈에 보이게 일하고 있다. G는 끝내 합류하지 못했다.
